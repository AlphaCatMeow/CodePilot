1. 项目定位

CodePilot 是一个多模型 AI Agent 桌面客户端。桌面外壳由 Electron 提供，前端、API 路由和 SSR 能力由 Next.js App Router 提供，本地持久化由 SQLite（better-sqlite3）提供，Agent 执行由多 Runtime 统一分发。

用户可见能力

聊天、模型切换、Provider 配置、MCP/Skills、远程 Bridge、媒体生成、项目文件树、Git 面板、终端、任务调度、上下文压缩、生成式 UI 和本地通知。

工程核心

所有 Agent 执行收敛到 streamClaude()，所有 Provider/Model 决策收敛到 provider-resolver.ts 与 providers/models API，所有 Runtime 输出统一为 SSE。

协作边界

当前项目规则明确：Codex 主要做审查、测试、复现分析、执行计划和文档；产品代码改动应交由 Claude Code 实施。本次产物仅新增文档。

主要来源：ARCHITECTURE.md、README.md、CLAUDE.md、src/lib/claude-client.ts、src/lib/runtime/*

3. 聊天主链路

桌面聊天的主链路由前端 startStream() 发起，后端 /api/chat 接收并完成会话锁、Provider/Runtime 解析、上下文拼装、MCP 选择、上下文压缩、Runtime 分发、SSE 转发和后台落库。 

MessageInput / ChatView
  → startStream(params) in src/lib/stream-session-manager.ts
  → fetch POST /api/chat
  → acquireSessionLock(session_id)
  → resolveRuntimeForSession(session)
  → resolveProviderForSession(intent, { runtime })
  → addMessage(user) unless autoTrigger
  → assembleContext({ entryPoint: "desktop" })
  → predictNativeRuntime(providerId)
       ├─ true  → loadAllMcpServers()
       └─ false → loadCodePilotMcpServers()
  → context estimate / auto compact
  → streamClaude(options)
  → runtime.stream(options)
  → SSE to browser
  → collectStreamResponse() tees stream and persists assistant result

关键不变量

不变量 实现位置 原因
发消息前必须先做 session-aware Provider/Runtime 解析 src/app/api/chat/route.ts 如果会话指向已删除 provider，要在写 user message 前返回 INVALID_SESSION_PROVIDER，不能静默切到默认 provider
每个 session 同时只能有一个运行中请求 acquireSessionLock()、renewSessionLock()、releaseSessionLock() 避免同一个会话并发写消息、并发复用 SDK session 或 Codex thread
用户第一轮发送会 lazy-seed runtime/model/provider /api/chat route 会话创建时可能没有固定模型或 provider；第一次真实用户发送后，当前解析结果成为会话事实
autoTrigger 不写 user message，不锁定 runtime /api/chat route 心跳、onboarding、技能展开等后台触发不应改变用户会话承诺
/compact 的反馈不入 DB /api/chat route 压缩提示是 UI artifact，不应成为下一轮模型上下文
客户端流状态不绑 React 生命周期 src/lib/stream-session-manager.ts 切换会话或组件重挂载后，正在运行的 SSE 仍可继续订阅

SSE 消费与持久化

前端的 consumeSSEStream() 负责把 SSE 事件转换成文本、thinking、tool use、tool result、权限请求、状态和结果。后端的 collectStreamResponse() 同时 tee 一份流用于落库：纯文本保存为字符串，包含 thinking/tool/media 的响应保存为结构化 JSON blocks。

工具结果中的 media block 会经 saveMediaToLibrary() 保存到本地媒体库，再把 base64 替换成本地路径和 mediaId。这个行为对图片生成、Codex 工具媒体回显、后续刷新恢复都很关键。

Provider 解析优先级

  2. session.provider_id
  3. default_provider_id setting
  4. active provider fallback
  5. env mode / legacy ANTHROPIC_* / app settings

resolveProviderForSession()
  └─ 在上述规则外增加 ghost provider 检查：

模型来源不是单一 catalog。聊天 picker、Settings 模型页、Provider 卡片、Runtime 兼容提示都围绕 /api/providers/models 和 provider_models 表协同工作。

模型发现 pipeline

Settings / Provider card refresh
  → POST /api/providers/[id]/discover-models
  → discoverModels(input)       // read-only upstream probe
  → build diff against provider_models
  → POST /api/providers/[id]/discover-models/apply
  → applyDiscoveryDiff()        // DB write, respects manual_* guards
入口 用途 安全边界
discoverModels() 读上游模型目录：OpenAI-compatible、Anthropic、OpenRouter、Gemini、Ollama 等 只读，不写 DB；8 秒默认超时；不回显 API key；fullModelIds 为权威列表，sampleModels 只给 UI 显示
classifyProvider() 判断是否能/该 probe Coding/Token Plan、OpenRouter 全量目录、图片 provider、OAuth/env-only 等会返回 unsupported，避免把非套餐 SKU 写进 DB
applyDiscoveryDiff() 把 discovery diff 写入 provider_models 不覆盖 manual_enabled/manual_hidden 用户意图；只自动调整 pristine/recommended/discovered/catalog row
seedCatalogModelsIfEmpty() 对不可发现 provider 物化 catalog 默认模型 幂等；已有 row 后不重复 seed
POST /api/providers/[id]/search-models OpenRouter 或可靠上游的搜索添加 搜索不写 DB；真正添加仍走 POST /api/providers/[id]/models

模型字段语义

字段 语义 使用注意
model_id CodePilot 内部/UI 选择值，可以是短别名 不要直接发给上游，除非确认无需映射
upstream_model_id 真正发给 API 的模型 ID toAiSdkConfig() 和 toClaudeCodeEnv() 都会优先映射这个值
display_name UI 展示名称 不代表上游可调用 ID
capabilities_json 模型能力，如 effort/thinking/context/media /api/providers/models 会把 effort/thinking 常用字段提升到顶层

6. Runtime 分发

Runtime 是“谁来跑 Agent loop”，Provider transport 是“怎么和模型服务说话”。这两个概念不要混在一起。前端和 API 层不直接知道具体 Runtime 的内部协议，只消费统一的 SSE 输出。

Runtime 注册 ID 核心文件 适用场景
Claude Code SDK Runtime claude-code-sdk src/lib/runtime/sdk-runtime.ts
src/lib/claude-client.ts		 通过 @anthropic-ai/claude-agent-sdk 子进程运行，复用 Claude Code 能力、settings、MCP、CLI 工具链
CodePilot Native Runtime native src/lib/runtime/native-runtime.ts
src/lib/agent-loop.ts
src/lib/ai-provider.ts		 基于 Vercel AI SDK 直接跑 agent loop，不要求安装 Claude Code CLI
Codex Runtime codex_runtime src/lib/codex/runtime.ts
src/lib/codex/app-server-*.ts
src/lib/codex/proxy/*		 通过 Codex app-server 运行，Codex Account 原生或 CodePilot provider proxy 路径

统一 Runtime 合约

所有 Runtime 实现 AgentRuntime：stream(options) 返回 SSE ReadableStream，interrupt(sessionId) 处理中断，isAvailable() 做可用性检测，dispose() 释放资源。

选择规则

streamClaude(options)
  1. codex_account provider → 必须走 Codex Runtime，缺失则报错
  2. session.runtime_pin 或 global agent_runtime 指向 codex_runtime → 优先 Codex
  3. openai-oauth provider → 默认 Native；若 Codex intent 明确则走 Codex proxy
  4. 非 Native-compatible transport → 尝试 SDK Runtime
  5. 否则 resolveRuntime(pin/global/auto)

resolveRuntime()
  0. Codex Runtime explicit/global
  1. 显式 override（session pin）优先，SDK 缺失时 fail closed
  2. cli_enabled=false → Native
  3. global agent_runtime 可用则使用，不可用则 fall through
  4. auto：Claude Code CLI 可用 → SDK，否则 Native
实践结论： 新增 Runtime 时只应实现 AgentRuntime 并在 src/lib/runtime/index.ts 注册。不要让 React 组件或普通 API route 直接依赖新 Runtime 的内部 SDK。

7. 请求构建复用点

项目里已经有多套请求构建器，分别对应 Claude Code SDK env、AI SDK model、Claude Code-compatible wire format、Codex Responses proxy、媒体 provider。不要在新功能里临时拼 body/header。

请求类型 复用入口 负责内容
Claude Code SDK 子进程请求 prepareSdkSubprocessEnv() → toClaudeCodeEnv() → SDK query() 隔离 ~/.claude、清理/注入 env、设置 model、cwd、permission、MCP settingSources、abortController
Native Runtime 模型实例 createModel() 按 AiSdkConfig.sdkType 创建 Anthropic/OpenAI/Google/Bedrock/Vertex/ClaudeCodeCompat model，套 middleware
Anthropic 第三方代理兼容请求 src/lib/claude-code-compat/request-builder.ts 把 AI SDK 请求转成 Claude Code/Anthropic-compatible wire format，适配 Kimi、GLM、OpenRouter Anthropic skin 等
Codex Responses proxy 请求 parse-request.ts、translate-input.ts、translate-tools.ts、unified-adapter.ts 解析 Codex app-server 入站 body，转 AI SDK messages/tools/providerOptions，再转回 Codex SSE 或非流式 JSON
OpenAI OAuth / Codex API ai-provider.ts 中 createOpenAI({ fetch }) 自定义 fetch 每次请求前 ensureTokenFresh()，重写到 chatgpt.com/backend-api/codex/responses，注入 Bearer 和 account id
Provider 连接测试 testProviderConnection() 构造最小 /v1/messages ping 请求，按 authStyle 设置 x-api-key 或 Authorization: Bearer
模型发现请求 discoverModels() 内各 probe 构造 /v1/models、Gemini models、Ollama tags 等请求，统一超时和脱敏

Claude Code-compatible 路径的判断

toAiSdkConfig() 会把非官方 Anthropic base URL 的 provider 路由到 claude-code-compat，因为很多第三方代理兼容 Claude Code wire format，但不完全兼容 @ai-sdk/anthropic 的 URL、beta headers 或 body 细节。OpenRouter Anthropic skin 也走同一路径。

8. 连接测试与诊断

连接测试和诊断已有独立路径，不能用“能否发起一次聊天”替代，因为聊天会引入 Runtime、MCP、session、上下文等变量。

Provider 连接测试

POST /api/providers/test
  → 从 body 取 providerId/presetKey/apiKey/baseUrl/protocol/authStyle/modelName
  → 如果 apiKey 是 "***xxxx" 或空，且 providerId 存在，则回填 DB 真 key
  → testProviderConnection()
       ├─ bedrock/vertex/env_only → SKIPPED
       ├─ openai-image/gemini-image → testMediaProviderConnection()
       └─ anthropic-compatible → POST {baseUrl}/v1/messages with max_tokens: 1
  → classifyError() 转用户可读错误和 recoveryActions
分支 行为 原因
authStyle=auth_token 发送 Authorization: Bearer <key> OpenRouter、GLM、MiniMax、Volcengine、Bailian 等 coding/token plan 常见
authStyle=api_key 发送 x-api-key: <key> Anthropic 官方、Kimi 等 API-key-only provider
bedrock / vertex / env_only 返回 SKIPPED 中性结果 这些 provider 依赖 IAM/OAuth/系统环境，不能用简单 HTTP ping 准确验证
媒体 provider 走媒体 provider 专用最小 probe 通用 Anthropic /v1/messages 对图片服务无意义，会产生假失败
第三方 Anthropic 缺 baseUrl 直接返回 MISSING_BASE_URL 避免测试误打官方 api.anthropic.com，产生假阳性

Provider Doctor

诊断中心由 src/lib/provider-doctor.ts、/api/doctor、/api/doctor/repair、/api/doctor/export 组成。已有文档 docs/handover/provider-error-doctor.md 记录了错误分类、Provider 生效、Auth 自动、5 探针诊断、runtime-log 脱敏和修复动作。新增 Provider 诊断时优先扩展这里，不要在设置页组件里直接写探针。

9. 上下文、MCP 与工具

上下文拼装

assembleContext() 是桌面聊天和 Bridge 的统一系统提示词拼装入口。它把稳定内容放在前缀，把每轮变化的内容放在后缀，以减少 prompt cache 失效。

层 内容 稳定性
1 Widget system prompt（仅 desktop） 静态前缀
2 session.system_prompt 静态前缀
3 Assistant workspace identity：soul/user/claude.md 静态前缀
4 memory hint 易变后缀
5 onboarding、heartbeat、progressive update、buddy instructions 易变后缀
6 Dashboard summary（desktop） 易变后缀
7 systemPromptAppend：技能、图片 agent mode 等每轮附加内容 易变后缀

MCP 加载

入口 用途 注意
loadCodePilotMcpServers() 只返回需要 CodePilot 特殊处理的 MCP server，例如含 ${...} DB 占位符的 env SDK Runtime 常用；其它 MCP 交给 Claude Code 自己从 settingSources 加载
loadAllMcpServers() 加载 user settings、CodePilot settings、项目 .mcp.json 合并后的所有 server Native Runtime 常用；它自己管理 MCP 连接
loadProjectMcpServers(projectCwd) 按用户实际项目目录加载 .mcp.json 不要用 process.cwd() 代表用户项目目录；桌面 app 里那通常是 Next server 目录
invalidateMcpCache() 添加、删除、切换 MCP server 后清缓存 确保设置页状态和下一轮运行读取一致

Native 工具组装

assembleTools() 是 Native Agent Loop 的工具入口。它合并核心 coding tools、CodePilot builtin tools、外部 MCP tools，并根据 mode 和 permissionContext 包装权限检查。 

assembleTools(options)
  ├─ createBuiltinTools()        // Read / Write / Edit / Bash / Glob / Grep / Agent / Skill
  ├─ getBuiltinTools()           // notification / memory / dashboard / media / cli-tools / widget 等
  ├─ buildMcpToolSet()           // 外部 MCP tools
  └─ wrapWithPermissions()
       ├─ safe_read tools 直接放行
       ├─ deny 直接返回 Permission denied
       └─ ask → createPermissionRequest() + registerPendingPermission()
权限原则： codepilot_* 工具不是天然安全。现在安全工具来自 mutation-level 分类和 PERMISSION_SAFE_TOOLS，未知或 mutating 工具默认进入权限包装。

11. Electron 本地能力

CodePilot 不是单纯浏览器应用。Electron 主进程启动或连接 Next server，管理窗口和托盘，暴露安全 preload API 给渲染进程，并处理终端、本地路径、通知、更新、导出等桌面能力。

能力 主进程实现 Preload API
内嵌 Next server utilityProcess.fork() 启动 standalone server；稳定端口 47823-47830，失败后动态端口 渲染进程通过正常 HTTP origin 访问
窗口与外链安全 BrowserWindow、外链拦截、window open handler 无直接 API；由 shell/openExternal 控制
安装 Claude Code / Git install:check-prerequisites、install:start、install:git window.electronAPI.install.*
本地路径和文件夹选择 shell:open-path、dialog:open-folder electronAPI.shell.openPath()、electronAPI.dialog.openFolder()、electronAPI.fs.getPathForFile()
Widget / Artifact 导出 隔离 BrowserWindow 截图，long-shot 使用 CDP Page.captureScreenshot electronAPI.exportWidgetAsPng()、electronAPI.artifact.exportLongShot()
集成终端 TerminalManager + terminal:create/write/resize/kill electronAPI.terminal.*
系统通知 notification:show、点击事件路由、后台 poller ack electronAPI.notification.show()、onClick()
系统代理解析 proxy:resolve 调 Chromium proxy config electronAPI.proxy.resolve(url)
日志与崩溃 Sentry Electron main、sanitized rotating log、main log path electronAPI.app.getLogPath()
端口设计： packaged app 优先使用稳定端口 47823-47830，是为了保持 localStorage origin 稳定。不要轻易改回随机端口，否则主题、最近模型等前端本地状态会随重启丢失。

14. 复用原则

  1. Provider 相关先找 resolver。 不要在组件或 route 中重新推断 authStyle、base URL、upstream model 或协议。
  2. 模型相关先找 /api/providers/models 和 provider_models。 不要把 catalog 当唯一来源，也不要绕过 hidden/manual 语义。
  3. Runtime 相关先找 streamClaude() 和 AgentRuntime。 UI 层只认 SSE，不认具体 SDK。
  4. 请求构建先找现有 adapter。 Claude Code SDK、AI SDK、ClaudeCodeCompat、Codex Responses、媒体 provider 都有各自构建路径。
  5. 连接测试不要走完整聊天。 连接测试使用 testProviderConnection() 或对应媒体 probe；诊断使用 Doctor。
  6. MCP 加载按 Runtime 分支。 SDK Runtime 尽量让 Claude Code 自己读 settingSources；Native Runtime 需要 CodePilot 合并后的完整 MCP。
  7. 上下文拼装只扩展 assembleContext()。 避免 Desktop 和 Bridge 两条入口提示词漂移。
  8. 用户可见状态必须有真实来源。 统计、能力、兼容性、上下文窗口、进度、badge、warning 都要能追到 source breadcrumb。
  9. 显式用户意图 fail closed。 session runtime pin、session provider、manual model visibility 都不应被全局默认或刷新流程悄悄覆盖。
  10. 文档和测试跟着风险走。 Provider/Runtime/DB/权限/Stream/MCP/Electron 属于高风险域，至少补 targeted 单测并跑 npm run test；UI 行为补 smoke/e2e。
